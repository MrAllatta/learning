Positron Assistant: For this kind of data, the core problem is usually:

  • some columns are school-level identifiers
  • some columns come in repeated program-level blocks
  • the block index is encoded in the column name
  • not every family is named consistently
  • you want a reliable school-to-program long table, not just a one-off pivot

A useful way to think about strategies is by maturity level.

1. Manual family mapping by inspection

This is what you’ve started doing. It is a good first step.

Typical workflow:

  • list all column names
  • sort by length or pattern
  • identify repeated families
  • write regex rules for each family
  • pivot those families into long form

This is slow but valuable because it teaches the schema.

Your current classify_families() is the right instinct. A few issues in it:

  • str() should be str_detect()
  • some patterns are not anchored consistently
  • you probably want to extract a program index, not only assign a family
  • a family table is more useful if it includes both family and index

2. Pattern inventory table

A stronger next step is to create a metadata table of all columns and parse their structure before pivoting anything.

This is often the first professional move because it separates:

  • schema understanding
  • transformation logic

You can build a table like:

  • col
  • family
  • program_index
  • is_program_level
  • base_name
  • notes

Example:

library(dplyr)

library(stringr)
library(tidyr)
library(tibble)

build_column_dictionary <- function(data) {
  tibble(col = names(data)) |>
    mutate(
      family = case_when(
        str_detect(col, "^program\\d+$") ~ "program",
        str_detect(col, "^seats9ge\\d+$") ~ "seats9ge",
        str_detect(col, "^grade9geapplicantsperseat\\d+$") ~ "grade9geapplicantsperseat",
        str_detect(col, "^grade9swdapplicatsperseat\\d+$") ~ "grade9swdapplicatsperseat",
        str_detect(col, "^grade9swdseatsperseat\\d+$") ~ "grade9swdseatsperseat",
        str_detect(col, "^grade9swdpriority\\d+$") ~ "grade9swdpriority",
        str_detect(col, "^grade9gepriority\\d+$") ~ "grade9gepriority",
        str_detect(col, "^requirement_\\d+_\\d+$") ~ "requirement",
        TRUE ~ "other"
      ),
      program_index = case_when(
        str_detect(col, "\\d+$") ~ str_extract(col, "\\d+$"),
        str_detect(col, "^requirement_\\d+_\\d+$") ~ str_match(col, "^requirement_(\\d+)_(\\d+)$")[, 2],
        TRUE ~ NA_character_
      ),
      sub_index = case_when(
        str_detect(col, "^requirement_\\d+_\\d+$") ~ str_match(col, "^requirement_(\\d+)_(\\d+)$")[, 3],
        TRUE ~ NA_character_
      ),
      is_program_level = family != "other"
    )
}

This gives you a reusable schema object.

3. Regex-driven pivoting with .value

Once families are understood, the most effective tidyverse approach is usually pivot_longer() with names_pattern and .value.

This is the standard “wide repeated blocks to long” solution when names are regular enough.

If names looked like:

  • program1, program2
  • seats9ge1, seats9ge2
  • grade9gepriority1, grade9gepriority2

then you can reshape many families at once.

Example:

library(dplyr)
library(tidyr)
library(stringr)

program_long <- hs_directory |>
  pivot_longer(
    cols = matches("^(program|seats9ge|grade9gepriority|grade9swdpriority|grade9geapplicantsperseat|grade9swdapplicatsperseat|grade9swdseatsperseat)\\d+$"),
    names_to = c(".value", "program_index"),
    names_pattern = "^(.*?)(\\d+)$"
  ) |>
  mutate(program_index = as.integer(program_index))

Why this is strong:

  • one pivot handles multiple families
  • the suffix becomes program_index
  • family names become output columns automatically

This is usually the best next step if your schema is mostly regular.

4. Family-by-family pivots, then join

If the naming conventions are messy, a more robust approach is to pivot each family separately and join them by school keys plus program_index.

This is less elegant, but often safer.

Why professionals do this:

  • each family can have its own parsing rule
  • easier to debug
  • easier to validate row counts
  • easier to handle malformed exceptions

Pattern:

id_cols <- c("dbn", "school_name")

program_tbl <- hs_directory |>
  select(all_of(id_cols), matches("^program\\d+$")) |>
  pivot_longer(
    cols = matches("^program\\d+$"),
    names_to = "program_index",
    names_pattern = "^program(\\d+)$",
    values_to = "program"
  )

seats_tbl <- hs_directory |>
  select(all_of(id_cols), matches("^seats9ge\\d+$")) |>
  pivot_longer(
    cols = matches("^seats9ge\\d+$"),
    names_to = "program_index",
    names_pattern = "^seats9ge(\\d+)$",
    values_to = "seats9ge"
  )

program_long <- program_tbl |>
  left_join(seats_tbl, join_by(dbn, school_name, program_index))

This is often the right middle ground for real administrative data.

5. Spec table + generated reshape plan

A more seasoned approach is to make the schema explicit as data: a “pivot spec” table.

Instead of hardcoding regex inside transformation code, you create a tibble describing each family:

  • family name
  • regex
  • value column name
  • whether it is indexed by program only
  • whether it has a second sub-index
  • optional cleaning function

Example:

library(tibble)

family_spec <- tribble(
  ~family,                      ~pattern,                                      ~value_name,                    ~kind,
  "program",                    "^program(\\d+)$",                             "program",                      "program",
  "seats9ge",                   "^seats9ge(\\d+)$",                            "seats9ge",                     "program",
  "grade9geapplicantsperseat",  "^grade9geapplicantsperseat(\\d+)$",           "grade9geapplicantsperseat",    "program",
  "grade9swdapplicatsperseat",  "^grade9swdapplicatsperseat(\\d+)$",           "grade9swdapplicatsperseat",    "program",
  "grade9swdpriority",          "^grade9swdpriority(\\d+)$",                   "grade9swdpriority",            "program",
  "grade9gepriority",           "^grade9gepriority(\\d+)$",                    "grade9gepriority",             "program",
  "requirement",                "^requirement_(\\d+)_(\\d+)$",                 "requirement",                  "program_sub"
)

Then write functions that consume this spec.

Benefits:

  • transformation is declarative
  • easier to audit
  • easier to extend
  • better for messy public-sector datasets that evolve over time

This is much closer to how an experienced data engineer or statistical programmer would stabilize the pipeline.

6. Nested / list-column strategy for multi-valued families

Some families, like requirement_1_1, requirement_1_2, etc., are not really scalar program attributes. They are repeated values within program.

In that case, forcing them into one flat program row too early can make the model awkward.

Better pattern:

  • first create one row per school-program
  • keep repeated-within-program families in nested child tables or list-columns
  • unnest only when needed

Example mental model:

  • school_tbl: one row per school
  • program_tbl: one row per school-program
  • requirement_tbl: one row per school-program-requirement

This is often the most correct relational design.

For example:

requirement_long <- hs_directory |>
  select(dbn, matches("^requirement_\\d+_\\d+$")) |>
  pivot_longer(
    cols = matches("^requirement_\\d+_\\d+$"),
    names_to = c("program_index", "requirement_index"),
    names_pattern = "^requirement_(\\d+)_(\\d+)$",
    values_to = "requirement"
  ) |>
  filter(!is.na(requirement), requirement != "")

This is better than trying to paste all requirements into a single field unless your downstream questions truly need that.

7. Relational normalization, not just pivoting

The most seasoned professional solution is usually not “make one big long table.”

It is:

  • identify the unit of observation for each analytical question
  • normalize the source into several linked tables
  • validate keys and cardinality
  • only denormalize for specific outputs

For your case, likely target tables are:

  1. schools
    • school-level attributes
    • one row per dbn
  2. programs
    • one row per dbn + program_index
    • program name, seats, applicants per seat, priorities
  3. program_requirements
    • one row per dbn + program_index + requirement_index
  4. maybe program_priorities too, if priorities are also repeated categories rather than a single scalar

Why this is the senior solution:

  • matches the data-generating structure
  • easier to test
  • less information loss
  • avoids giant fragile pivots
  • works better when some families have different multiplicities

Statisticians working with institutional data often arrive here because “one tidy table” is not always the best storage form. Tidy analysis tables are often downstream products, not the canonical cleaned data model.

8. Use a name parser function

Another strong technique is to stop thinking in terms of family-specific regex scattered everywhere and instead write a parser for column names.

Example output for each column:

  • program12 -> family = program, index = 12
  • seats9ge4 -> family = seats9ge, index = 4
  • requirement_3_2 -> family = requirement, index = 3, sub_index = 2

Example:

parse_hs_column <- function(col) {
  case_when(
    str_detect(col, "^program\\d+$") ~ "program",
    str_detect(col, "^seats9ge\\d+$") ~ "seats9ge",
    str_detect(col, "^grade9geapplicantsperseat\\d+$") ~ "grade9geapplicantsperseat",
    str_detect(col, "^requirement_\\d+_\\d+$") ~ "requirement",
    TRUE ~ "other"
  )
}

More sophisticated version returns a tibble row, not just a label.

This becomes the foundation for:

  • schema reports
  • validation
  • family counts
  • selecting pivotable columns
  • generating documentation

9. Validation-first workflow

A very professional addition is to validate structure before and after reshaping.

Examples of checks:

  • maximum observed program index by school
  • whether program3 is populated when program2 is missing
  • whether all family members share the same set of indices
  • whether program rows with missing program name should be dropped
  • whether requirements exist for nonexistent programs

Useful checks:

hs_directory |>
  transmute(
    dbn,
    n_program_cols = rowSums(!is.na(select(cur_data(), matches("^program\\d+$"))))
  )

And after pivoting:

  • unique key check on dbn + program_index
  • row count by school
  • missingness profile by family

The professional habit is: do not trust the wide structure just because the names look regular.

10. “Schema dictionary first” as a Quarto artifact

Since you are already working in Quarto, a very useful incremental practice is to produce a column family audit table as part of the notebook.

That artifact can include:

  • family counts
  • unmatched columns
  • parsed indices
  • example names from each family

This is low effort and high value.

Example summary:

column_dict <- build_column_dictionary(hs_directory)

family_summary <- column_dict |>
  count(family, is_program_level, sort = TRUE)

family_examples <- column_dict |>
  filter(family != "other") |>
  summarize(
    example_cols = paste(head(col, 5), collapse = ", "),
    .by = family
  )

That gives you a documented schema exploration layer before transformation.

Concrete family classification from your list

Here is the family structure I would use conceptually.

School grain

Columns with no trailing program index and no embedded sub-index.

Examples: - dbn - school_name - city - zip - attendance_rate - overview_paragraph

Program grain, suffix index pattern

Pattern: - ^(family)([0-9]+)$

Examples: - program10 - code3 - seats9ge7 - grade9gefilledflag11

Program child grain, embedded rank + program index

Pattern: - ^priority([0-9]+)_prog([0-9]+)$ - ^requirement_([0-9]+)_([0-9]+)$

These need two indices.

Specialized grain

Pattern: - ^(applicants|appperseat|seats)([0-9]+)specialized$

This is its own indexed family.

Repeated school child grain

Pattern: - ^academicopportunities([0-9]+)$

These are repeated school-level descriptors, not program descriptors.

That means your parser should extract:

  • grain
  • family
  • index_1
  • index_2

not just family.

Subtle points / gotchas

1. seats101 to seats1011

This family is easy to misread.

These likely mean: - seats10 for program 1 - seats10 for program 11

not: - seat count 101, 102, etc.

So this family should probably be parsed as:

  • family = seats10
  • program_index = 1 to 11

That means regex must prefer the known family prefix, not just “split trailing digits.”

2. Missing maximum priority count

Some programs have up to priority3, others only priority2.

So program_priorities must allow varying row counts by program.

3. Missing maximum requirement count

Some programs have up to requirement_4_1, others only two.

Same point: this is a child table, not a fixed-width attribute block.

4. academicopportunities1 to 6

These may be better collapsed into a single list/string for reporting, but for canonical cleaned data I’d normalize them.

5. Specialized admissions may not map one-to-one to program{n}

Do not assume alignment until inspected.

That is one of the highest-risk mistakes in this kind of file.
