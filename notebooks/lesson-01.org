** The BigQuery Storage Warning

The warning says the *BigQuery Storage API* client library isn't installed, so the library falls back to the *REST API* to download results.

*** Two ways to fetch query results

| Method                     | Library needed                              | Speed                                          |
|----------------------------+---------------------------------------------+------------------------------------------------|
| *Storage Read API*         | =google-cloud-bigquery-storage= + =pyarrow= | Fast — streams data in columnar (Arrow) format |
| *REST endpoint* (fallback) | nothing extra                               | Slower — JSON over HTTPS, row by row           |

*** Why it doesn't matter here

For small results (10 rows) the REST fallback is perfectly fine. You'd only care about the Storage API when pulling *large result sets* (millions of rows) where the columnar streaming is significantly faster.

*** To silence the warning

Install the optional extras:

#+begin_src bash
pip install google-cloud-bigquery[bigquery_storage]
# or explicitly:
pip install google-cloud-bigquery-storage pyarrow
#+end_src

After that, =bigquery= will detect the library and use the faster path automatically — and the warning disappears.

* Python source blocks versus Jupyter
What benefits would I see from using a jupyter kernel rather than a python virtual environment? Are they mutually exclusive?

** They are not mutually exclusive

A Jupyter kernel /is/ a Python process — typically the Python from a virtualenv (or conda env). You register a venv as a kernel:

#+begin_src bash
  source .venv/bin/activate
  pip install ipykernel
  python -m ipykernel install --user --name my-project
#+end_src

Then Jupyter uses that venv's Python.

* What you gain with a Jupyter kernel in Emacs (via =ein= or =jupyter= package)

| Feature                    | Python session (current) | Jupyter kernel         |
|----------------------------+--------------------------+------------------------|
| Rich output (plots, HTML)  | No — text only           | Yes — inline images    |
| DataFrame display          | =print(df)=              | Pretty HTML table      |
| Persistent state indicator | Opaque                   | Cell execution counter |
| Interrupt/restart          | Clunky                   | Clean kernel controls  |
| Variable inspector         | No                       | Available via plugins  |
| Sharing notebooks          | Export manually          | Native =.ipynb= format |

You get:
- Clean separation of code and output
- Version-controllable figures (or gitignored, your choice)
- No Jupyter dependency
- Works with =ox-hugo=, Quarto, HTML export unchanged

The only thing you /don't/ get is figures appearing /inline as output/ of the block itself — you place them manually below. A minor inconvenience.
